# Notes Flow

Telegram бот для управления заметками и задачами с AI-саммаризацией. Бот собирает сообщения из личных чатов, группирует связанные сообщения и автоматически создаёт задачи/заметки с помощью локальной AI-модели.

## Требования

- Python 3.12+
- Локальная AI-модель (OpenAI-compatible API)

## Установка

1. Создайте виртуальное окружение:
```bash
python3 -m venv venv
```

2. Активируйте окружение:
```bash
source venv/bin/activate
```

3. Установите зависимости:
```bash
pip install -r requirements.txt
```

4. Установите типы для PyYAML (опционально, для mypy):
```bash
pip install types-PyYAML
```

5. Скопируйте `.env.example` в `.env` и настройте переменные окружения.

## Конфигурация

Создайте файл `.env` в корне проекта со следующими переменными:

```env
TELEGRAM_BOT_TOKEN=your_telegram_bot_token_here
OLLAMA_BASE_URL=http://localhost:11434
OLLAMA_MODEL=llama3
DEFAULT_SUMMARIZE_DELAY=300
```

### Параметры конфигурации

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| `TELEGRAM_BOT_TOKEN` | Токен вашего бота от @BotFather | (обязательно) |
| `OLLAMA_BASE_URL` | URL AI API (OpenAI-compatible) | `http://localhost:8080` |
| `OLLAMA_MODEL` | Название модели для саммаризации | `unsloth/Qwen3.5-35B-A3B` |
| `DEFAULT_SUMMARIZE_DELAY` | Задержка перед саммаризацией в секундах | `300` (5 минут) |

**Отправка сообщений:**

### Резервное копирование и восстановление

**Создание бэкапа:**
```
/backup
```
Бот создаст ZIP-архив со всеми вашими данными (задачи, заметки, инбокс, архивы) и отправит его вам как документ.

**Восстановление из бэкапа:**
```
/restore
[Загрузите ZIP-файл как документ]
```
Бот покажет предварительный просмотр с количеством задач, заметок и сообщений перед восстановлением.

**Важно:**

### Автоматические ежедневные бэкапы

Бот автоматически создает ежедневные бэкапы ваших данных.

**Как это работает:**
- Ночная архивация и бэкап запускаются **каждый день в 02:00**
- Если файлы не изменились с последнего бэкапа, вы получите уведомление "Данные не изменились"
- Если есть изменения, вам придет полный бэкап в виде ZIP-файла
- Бэкапы хранятся только последние (история не сохраняется)

**Важно:**
- Бэкап запускается **только после ночной архивации** (не после ручной команды `/archive`)
- Бэкап НЕ запускается после саммаризации (автоматической или ручной)

**Примеры уведомлений:**

Без изменений:
```
ℹ️ Данные не изменились за последние 24 часа

Бэкап не создан.
```

С изменениями:
```
📦 Ежедневный бэкап создан
[backup.zip]
```

**Настройка:**
- Автоматические бэкапы включены для всех пользователей
- Отключить нельзя (это важная функция безопасности)

**Важно:**
- Перед восстановлением создаётся автоматический бэкап текущих данных
- Подтверждение требуется перед заменой данных
- При ошибке восстановления данные откатываются к состоянию до restore

### Устранение неполадок

**Ошибка "Нет данных для бэкапа"**
- Убедитесь, что у вас есть хотя бы одно сообщение в инбоксе или созданная задача/заметка

**Ошибка восстановления**
- Проверьте, что ZIP-файл не повреждён
- Убедитесь, что файл был создан командой `/backup` этого же бота
- При ошибке данные автоматически откатываются к состоянию до restore

**ZIP-файл не открывается**
- Используйте стандартный архиватор (7-Zip, WinRAR, встроенный в ОС)
- Файл создан по стандарту ZIP и совместим со всеми архиваторами

### ⚠️ Известная проблема: Docker Compose 'ContainerConfig' error

При обновлении образа через `docker-compose up -d` может возникнуть ошибка:

```
KeyError: 'ContainerConfig'
```

**Причина**: Несовместимость версий docker-compose с обновлённым образом контейнера.

**Решение** (правильный передеплой):

```bash
# 1. Полностью удалите старый контейнер и образ
docker-compose -f docker-compose.local.yml down --rmi all

# 2. Пересоберите образ без кэша (важно для применения изменений)
docker-compose -f docker-compose.local.yml build --no-cache

# 3. Запустите обновлённый контейнер
docker-compose -f docker-compose.local.yml up -d
```

**Или один командой** (рекомендуется):

```bash
docker-compose -f docker-compose.local.yml down --rmi all && \
docker-compose -f docker-compose.local.yml build --no-cache && \
docker-compose -f docker-compose.local.yml up -d
```

**Важно**: Всегда используйте `--no-cache` при пересборке, чтобы изменения в коде были применены.

# 2. Создайте новый контейнер
docker-compose -f docker-compose.local.yml up -d
```

**Альтернативный способ** (если первый не работает):
```bash
# Удалите контейнер вручную
docker rm -f notes-flow-local

# Запустите заново
docker-compose -f docker-compose.local.yml up -d
```

### Pre-deploy проверки

Перед развёртыванием новой версии автоматически запускаются тесты и проверка покрытия кода:

```bash
./pre-deploy.sh
```

**Важно:**
- Скрипт запускает все тесты (`pytest`) с измерением покрытия (`coverage`)
- Проверяет, что покрытие кода >= 80% (настраивается через `COVERAGE_THRESHOLD`)
- Блокирует развёртывание, если тесты падают или покрытие ниже порога
- Старый контейнер продолжает работать во время проверок

**Настройка порога покрытия:**
```bash
COVERAGE_THRESHOLD=90 ./pre-deploy.sh
```

### Docker Compose test workflow

Для локального тестирования перед развёртыванием:

```bash
docker-compose -f docker-compose.yml -f docker-compose.test.yml up --build test
```

Это запустит:
1. Старый контейнер с ботом (если ещё не запущен)
2. Тестовый контейнер, который:
   - Собирает покрытие
   - Запускает все тесты
   - Проверяет порог покрытия
3. Заменяет старый контейнер только если все проверки прошли
